# Java AI Training Project

Учебный репозиторий для практического освоения Java-разработки с использованием ИИ-агентов.

## Цель

Основная задача студента - реализовать техническое задание, используя современные ИИ-агенты как основной инструмент разработки.

Допускается использование любых ИИ-агентов: Codex, Kilo Code, Windsurf, Cursor, Claude Code, GitHub Copilot и других.

Основной акцент сделан на умении грамотно формулировать задачи ИИ, проверять результат и доводить проект до рабочего состояния.

---

## Структура проекта

```
docs/
```

Папка с подробным техническим заданием, которое необходимо реализовать.
Задача: Разработать админ панель.


```
infra/
```

Инфраструктура проекта.

Содержит Docker Compose и все необходимые файлы для автоматического запуска локальной базы данных PostgreSQL.

После запуска инфраструктуры автоматически:

* создается пользователь базы данных;
* создается база данных;
* создается схема приложения;
* выполняются миграции;
* загружаются тестовые данные.

Дополнительная настройка базы данных не требуется.

---

## Порядок работы

1. Запустить инфраструктуру из каталога `infra`.
2. Дождаться успешного завершения инициализации базы данных.
3. Реализовать проект, используя ИИ-агента.
4. Проверить соответствие результата требованиям технического задания.

Команды запуска инфраструктуры:

```bash
cd infra
docker compose up -d
docker compose ps
docker compose logs liquibase
```

PostgreSQL должен остаться запущенным, а контейнер Liquibase после успешного применения миграций должен завершиться без ошибки.

## Наставление студентам

Вы уже решали эту задачу ранее. Техническое задание в основном не изменилось: вместо локального MySQL и ручного выполнения SQL-скриптов теперь используется PostgreSQL, Liquibase и Docker Compose, чтобы база данных поднималась автоматически и одинаково у всех. В остальном логика приложения, роли, экраны, REST API, валидация и критерии приемки остались теми же.

ТЗ в каталоге `docs` расписано очень подробно. Его достаточно для точного решения задачи без самостоятельного изменения постановки. Ваша задача сейчас - не "угадать", как должно работать приложение, а научиться управлять ИИ-агентом так, чтобы он последовательно реализовал требования.

Попробуйте разные стратегии:

* решать задачу по шагам: сначала инфраструктура и конфигурация, затем сущности и БД, потом Security, REST API, UI и финальная проверка;
* решать задачу крупным запросом целиком, а затем просить агента проверить результат по каждому разделу ТЗ;
* комбинировать подходы: дать агенту общий контекст, но просить реализовывать и коммитить небольшими законченными частями.

Хорошая практика - просить агента не только писать код, но и сверять результат с критериями приемки. Если агент предлагает изменить требования, сначала проверьте ТЗ. Требования менять нельзя без явного согласования.

Ведите prompt log: сохраняйте важные промпты, ответы агента, найденные ошибки и решения. Это поможет понять, какие формулировки работают лучше, а какие приводят к неполному или неверному результату.

Делайте частые локальные коммиты. Коммит после каждого законченного этапа позволяет спокойно экспериментировать с промптами, откатываться к рабочему состоянию и сравнивать разные варианты решения.

Фиксируйте промежуточные результаты: что уже реализовано, какие проверки прошли, какие ошибки остались, какие вопросы нужно уточнить. Это дисциплинирует работу с ИИ и помогает не потерять контекст.

Главная цель упражнения - поэкспериментировать с промтингом и промпт-инжинирингом на реальной инженерной задаче. Чем точнее вы ставите задачу, тем легче проверить результат и довести проект до рабочего состояния.

## Важно

Технические задания являются единственным источником требований.

Реализация должна полностью соответствовать требованиям, указанным в документации.

При возникновении неоднозначностей необходимо уточнять требования у ИИ-агента или преподавателя, а не изменять постановку задачи самостоятельно.
