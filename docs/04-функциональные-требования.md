# 4. Функциональные требования

## 4.1. Аутентификация

### FR-1. Страница входа

| ID | Требование |
|---|---|
| FR-1.1 | Система должна предоставлять страницу входа по URL `/auth/login`, доступную без аутентификации. |
| FR-1.2 | На странице входа должны быть поля для ввода email (label: "Email address") и пароля (label: "Password"). |
| FR-1.3 | На странице должна быть кнопка "Sign in" для отправки формы. |
| FR-1.4 | Форма входа должна отправляться методом POST на URL `/process_login`. |
| FR-1.5 | При неверных учётных данных система должна перенаправлять на `/auth/login?error` и отображать сообщение "Неправильные имя или пароль" красным цветом. |
| FR-1.6 | Аутентификация должна производиться по email (в качестве username) и паролю. |
| FR-1.7 | Страница входа должна быть стилизована с использованием Bootstrap (фон WhiteSmoke, центрированная форма шириной 300px). |

### FR-2. Обработка успешного входа

| ID | Требование |
|---|---|
| FR-2.1 | После успешной аутентификации система должна перенаправлять пользователя в зависимости от его роли. |
| FR-2.2 | Если пользователь имеет роль `ROLE_ADMIN`, перенаправление должно осуществляться на `/admin`. |
| FR-2.3 | Если пользователь имеет только роль `ROLE_USER` (не имеет `ROLE_ADMIN`), перенаправление должно осуществляться на `/user`. |

### FR-3. Выход из системы

| ID | Требование |
|---|---|
| FR-3.1 | Система должна поддерживать выход из системы по URL `/logout` (метод POST). |
| FR-3.2 | После выхода система должна перенаправлять пользователя на страницу входа `/auth/login`. |
| FR-3.3 | На всех защищённых страницах должна быть кнопка "Logout" в верхней панели. |

## 4.2. Управление пользователями (Администратор)

### FR-4. Панель администратора

| ID | Требование |
|---|---|
| FR-4.1 | Система должна предоставлять панель администратора по URL `/admin`, доступную только пользователям с ролью `ROLE_ADMIN`. |
| FR-4.2 | Панель должна содержать левое навигационное меню с двумя пунктами: "Admin" и "User". |
| FR-4.3 | При выборе пункта "Admin" должна отображаться административная панель с двумя вкладками: "Users table" и "Add user". |
| FR-4.4 | При выборе пункта "User" должна отображаться информация о текущем авторизованном пользователе. |

### FR-5. Таблица пользователей

| ID | Требование |
|---|---|
| FR-5.1 | На вкладке "Users table" должна отображаться таблица всех пользователей системы. |
| FR-5.2 | Таблица должна содержать колонки: ID, First Name, Last Name, Age, Email, Role, Edit, Delete. |
| FR-5.3 | Данные таблицы должны загружаться асинхронно через REST API при загрузке страницы. |
| FR-5.4 | В колонке "Role" роли должны отображаться через запятую (например, "ROLE_USER, ROLE_ADMIN"). |
| FR-5.5 | В колонке "Edit" для каждой строки должна быть кнопка "Edit" (синего цвета, класс `btn-info`). |
| FR-5.6 | В колонке "Delete" для каждой строки должна быть кнопка "Delete" (красного цвета, класс `btn-danger`). |

### FR-6. Создание пользователя

| ID | Требование |
|---|---|
| FR-6.1 | На вкладке "Add user" должна быть форма создания нового пользователя. |
| FR-6.2 | Форма должна содержать поля: First name, Last name, Age, Email, Password, Role (множественный выбор). |
| FR-6.3 | Поле "Role" должно представлять собой multiple select с вариантами: USER (значение `ROLE_USER`) и ADMIN (значение `ROLE_ADMIN`). |
| FR-6.4 | Форма должна отправляться методом POST на REST API `/users/` в формате JSON. |
| FR-6.5 | После успешного создания пользователя таблица пользователей должна автоматически обновиться. |
| FR-6.6 | После успешного создания должна происходить автоматическая активация вкладки "Users table". |
| FR-6.7 | Кнопка отправки формы должна иметь текст "Add new user" и зелёный цвет (`btn-success`). |

### FR-7. Редактирование пользователя

| ID | Требование |
|---|---|
| FR-7.1 | При нажатии кнопки "Edit" в строке таблицы должно открываться модальное окно "Edit user". |
| FR-7.2 | Перед открытием модального окна система должна асинхронно загрузить данные пользователя по его ID. |
| FR-7.3 | Модальное окно должно содержать поля: ID (скрытое), First name, Last name, Age, Email, Password, Role (multiple select). |
| FR-7.4 | Все поля в модальном окне редактирования должны быть предзаполнены данными пользователя. |
| FR-7.5 | В поле выбора ролей должны быть предварительно выделены текущие роли пользователя. |
| FR-7.6 | Модальное окно должно иметь кнопку "Close" (серая, `btn-secondary`) для закрытия без сохранения. |
| FR-7.7 | Модальное окно должно иметь кнопку "Edit" (синяя, `btn-primary`) для сохранения изменений. |
| FR-7.8 | При нажатии "Edit" данные должны отправляться методом PATCH на REST API `/users/{id}` в формате JSON. |
| FR-7.9 | После успешного обновления модальное окно должно закрыться, а таблица пользователей — обновиться. |
| FR-7.10 | При ошибке обновления должно отображаться сообщение "Error updating user" через `alert()`. |

### FR-8. Удаление пользователя

| ID | Требование |
|---|---|
| FR-8.1 | При нажатии кнопки "Delete" в строке таблицы должно открываться модальное окно "Delete user". |
| FR-8.2 | Перед открытием модального окна система должна асинхронно загрузить данные пользователя по его ID. |
| FR-8.3 | Модальное окно должно содержать поля: ID (скрытое), First name, Last name, Age, Email, Password, Role (multiple select). |
| FR-8.4 | Все поля в модальном окне удаления должны быть **недоступны для редактирования** (атрибут `disabled`). |
| FR-8.5 | Поля должны быть предзаполнены данными пользователя. |
| FR-8.6 | В поле выбора ролей должны быть предварительно выделены текущие роли пользователя (select disabled). |
| FR-8.7 | Модальное окно должно иметь кнопку "Close" (серая, `btn-secondary`) для закрытия без удаления. |
| FR-8.8 | Модальное окно должно иметь кнопку "Delete" (синяя, `btn-primary`) для подтверждения удаления. |
| FR-8.9 | При нажатии "Delete" должен отправляться запрос методом DELETE на REST API `/users/{id}`. |
| FR-8.10 | После успешного удаления модальное окно должно закрыться, а таблица пользователей — обновиться. |
| FR-8.11 | При ошибке удаления должно отображаться сообщение "Error updating user" через `alert()`. |

## 4.3. Просмотр информации о пользователе

### FR-9. Страница пользователя

| ID | Требование |
|---|---|
| FR-9.1 | Система должна предоставлять страницу пользователя по URL `/user`, доступную всем аутентифицированным пользователям. |
| FR-9.2 | Страница должна отображать информацию о текущем авторизованном пользователе. |
| FR-9.3 | Левое навигационное меню должно содержать один пункт: "User". |
| FR-9.4 | На странице должна быть таблица с колонками: ID, First Name, Last Name, Age, Email, Role. |
| FR-9.5 | В колонке "Role" должны отображаться названия ролей (ADMIN и/или USER). |
| FR-9.6 | Данные пользователя должны передаваться с сервера через модель Thymeleaf (атрибут `authUser`). |

## 4.4. Информация об авторизованном пользователе (верхняя панель)

### FR-10. Шапка страницы

| ID | Требование |
|---|---|
| FR-10.1 | На всех защищённых страницах (admin, user) сверху должна быть панель с серым фоном (`darkgrey`). |
| FR-10.2 | В панели должен отображаться email текущего пользователя и текст "with roles:". |
| FR-10.3 | После текста "with roles:" должны отображаться названия ролей: "ADMIN" (если есть `ROLE_ADMIN`) и/или "USER" (если есть `ROLE_USER`). |
| FR-10.4 | В правой части панели должна быть кнопка "Logout" (ссылка, класс `btn-link text-muted`). |
| FR-10.5 | Кнопка "Logout" должна отправлять POST-запрос на `/logout`. |

## 4.5. REST API

### FR-11. Получение всех пользователей

| ID | Требование |
|---|---|
| FR-11.1 | Система должна предоставлять REST API эндпоинт `GET /users/` для получения списка всех пользователей. |
| FR-11.2 | Ответ должен быть в формате JSON массива объектов пользователей. |
| FR-11.3 | Доступ к эндпоинту должны иметь все аутентифицированные пользователи. |

### FR-12. Получение пользователя по ID

| ID | Требование |
|---|---|
| FR-12.1 | Система должна предоставлять REST API эндпоинт `GET /users/{id}` для получения пользователя по ID. |
| FR-12.2 | Ответ должен быть в формате JSON объекта пользователя. |
| FR-12.3 | Если пользователь с указанным ID не найден, должен возвращаться HTTP 404 с телом ошибки. |
| FR-12.4 | Доступ к эндпоинту должны иметь все аутентифицированные пользователи. |

### FR-13. Создание пользователя (REST)

| ID | Требование |
|---|---|
| FR-13.1 | Система должна предоставлять REST API эндпоинт `POST /users/` для создания пользователя. |
| FR-13.2 | Запрос должен приниматься в формате JSON (UserDTO). |
| FR-13.3 | Доступ к эндпоинту должны иметь только пользователи с ролью `ROLE_ADMIN`. |
| FR-13.4 | При успешном создании должен возвращаться HTTP 200. |
| FR-13.5 | При ошибке валидации должен возвращаться HTTP 400 с описанием ошибок. |
| FR-13.6 | Пароль должен храниться в зашифрованном виде (BCrypt). |

### FR-14. Обновление пользователя (REST)

| ID | Требование |
|---|---|
| FR-14.1 | Система должна предоставлять REST API эндпоинт `PATCH /users/{id}` для обновления пользователя. |
| FR-14.2 | Запрос должен приниматься в формате JSON (UserDTO). |
| FR-14.3 | Доступ к эндпоинту должны иметь только пользователи с ролью `ROLE_ADMIN`. |
| FR-14.4 | При успешном обновлении должен возвращаться HTTP 200. |
| FR-14.5 | При ошибке валидации должен возвращаться HTTP 400 с описанием ошибок. |
| FR-14.6 | Пароль при обновлении должен храниться в зашифрованном виде (BCrypt). |

### FR-15. Удаление пользователя (REST)

| ID | Требование |
|---|---|
| FR-15.1 | Система должна предоставлять REST API эндпоинт `DELETE /users/{id}` для удаления пользователя. |
| FR-15.2 | Доступ к эндпоинту должны иметь только пользователи с ролью `ROLE_ADMIN`. |
| FR-15.3 | При успешном удалении должен возвращаться HTTP 200. |

## 4.6. Управление ролями

### FR-16. Роли

| ID | Требование |
|---|---|
| FR-16.1 | Система должна поддерживать две предопределённые роли: `ROLE_USER` и `ROLE_ADMIN`. |
| FR-16.2 | Пользователь может иметь одну или несколько ролей одновременно. |
| FR-16.3 | Роли должны храниться в отдельной таблице базы данных. |
| FR-16.4 | Связь между пользователями и ролями — многие-ко-многим (M:N). |
