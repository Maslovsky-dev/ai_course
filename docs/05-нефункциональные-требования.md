# 5. Нефункциональные требования

## 5.1. Производительность

| ID | Требование |
|---|---|
| NFR-1.1 | Загрузка списка пользователей через REST API должна выполняться с использованием EntityGraph для предотвращения проблемы N+1 запросов при загрузке ролей. |
| NFR-1.2 | Все методы сервисного слоя, обращающиеся к БД, должны быть транзакционными. |
| NFR-1.3 | Загрузка ролей пользователя должна использовать ленивую загрузку (FetchType.LAZY) с применением EntityGraph для оптимизации. |

## 5.2. Безопасность

| ID | Требование |
|---|---|
| NFR-2.1 | Пароли должны храниться в базе данных в зашифрованном виде с использованием алгоритма BCrypt. |
| NFR-2.2 | CSRF-защита должна быть отключена (для поддержки REST API через AJAX). |
| NFR-2.3 | Должна быть настроена CORS-политика: разрешены запросы с origin `http://localhost:8080`. |
| NFR-2.4 | Разрешённые HTTP-методы для CORS: GET, POST, PATCH, DELETE, OPTIONS. |
| NFR-2.5 | Разрешённые заголовки для CORS: Authorization, Content-Type, Accept, X-XSRF-TOKEN. |
| NFR-2.6 | CORS должен поддерживать учётные данные (allowCredentials = true). |
| NFR-2.7 | Доступ к статическим JavaScript-файлам (`/js/**`) и страницам входа/ошибки должен быть открыт без аутентификации. |
| NFR-2.8 | Все остальные запросы должны требовать аутентификации. |
| NFR-2.9 | Удаление, создание и обновление пользователей (POST, PATCH, PUT, DELETE на `/users/**`) должно быть доступно только роли ADMIN. |
| NFR-2.10 | Страница `/admin` должна быть доступна только роли ADMIN. |
| NFR-2.11 | Учётные записи пользователей должны всегда быть активными: accountNonExpired, accountNonLocked, credentialsNonExpired, enabled — все true. |

## 5.3. Надёжность и обработка ошибок

| ID | Требование |
|---|---|
| NFR-3.1 | При запросе несуществующего пользователя по ID должно возвращаться HTTP 404 с JSON-ответом, содержащим сообщение и timestamp. |
| NFR-3.2 | При ошибках валидации при создании/обновлении пользователя должно возвращаться HTTP 400 с JSON-ответом, содержащим список ошибок и timestamp. |
| NFR-3.3 | Ошибки на клиенте (JavaScript) должны логироваться в консоль через `console.error`. |
| NFR-3.4 | При сетевых ошибках на клиенте должно отображаться уведомление через `alert()`. |

## 5.4. Совместимость

| ID | Требование |
|---|---|
| NFR-4.1 | Приложение должно работать на JDK 21. |
| NFR-4.2 | Приложение должно использовать PostgreSQL 16 в качестве СУБД. |
| NFR-4.3 | Веб-интерфейс должен корректно отображаться в современных браузерах с поддержкой ES6 и Fetch API. |
| NFR-4.4 | UI должен быть адаптивным (responsive) с использованием Bootstrap grid system. |

## 5.5. Масштабируемость

| ID | Требование |
|---|---|
| NFR-5.1 | Приложение должно быть stateless (сессия управляется Spring Security). |
| NFR-5.2 | Архитектура должна позволять горизонтальное масштабирование базы данных. |

## 5.6. Развёртывание

| ID | Требование |
|---|---|
| NFR-6.1 | Приложение должно упаковываться в исполняемый JAR-файл с встроенным Tomcat. |
| NFR-6.2 | Приложение должно запускаться командой `mvn spring-boot:run` или `java -jar`. |
| NFR-6.3 | Порт по умолчанию — 8080. |
| NFR-6.4 | Локальная инфраструктура PostgreSQL 16 и отдельный контейнер Liquibase должны запускаться из каталога `infra` командами `cd infra` и `docker compose up -d`. |
| NFR-6.5 | Hibernate должен работать в режиме `validate` (проверка схемы без автоматического создания/изменения). |
| NFR-6.6 | Docker Compose должен автоматически создавать пользователя PostgreSQL, базу данных, схему приложения `admin_panel`, запускать Liquibase, применять все миграции и заполнять базу тестовыми данными. |
| NFR-6.7 | Java-приложение не должно подключать Liquibase или запускать миграции средствами Spring Boot. |
| NFR-6.8 | Java-приложение должно явно использовать схему приложения `admin_panel`; использование схемы `public` не допускается. |

## 5.7. Логирование

| ID | Требование |
|---|---|
| NFR-7.1 | SQL-запросы Hibernate должны логироваться (show_sql = true). |
| NFR-7.2 | Клиентские ошибки JavaScript должны выводиться в консоль браузера. |
