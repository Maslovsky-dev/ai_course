# 9. Роли пользователей и безопасность

## 9.1. Роли пользователей

### 9.1.1. Определение ролей

Система поддерживает две роли:

| Роль | Идентификатор | Описание |
|---|---|---|
| **Пользователь** | `ROLE_USER` | Базовая роль. Доступ только к просмотру собственной информации. |
| **Администратор** | `ROLE_ADMIN` | Расширенная роль. Полный доступ к управлению пользователями и просмотру собственной информации. |

### 9.1.2. Матрица доступа

| Ресурс / Действие | ROLE_USER | ROLE_ADMIN | Неаутентифицированный |
|---|---|---|---|
| `/auth/login` (GET) | ✅ | ✅ | ✅ |
| `/error` | ✅ | ✅ | ✅ |
| `/js/**` (статические JS) | ✅ | ✅ | ✅ |
| `/admin` (GET) | ❌ (403) | ✅ | → redirect to `/auth/login` |
| `/user` (GET) | ✅ | ✅ | → redirect to `/auth/login` |
| `GET /users/` | ✅ | ✅ | ❌ (403) |
| `GET /users/{id}` | ✅ | ✅ | ❌ (403) |
| `POST /users/**` | ❌ (403) | ✅ | → redirect to `/auth/login` |
| `PATCH /users/**` | ❌ (403) | ✅ | → redirect to `/auth/login` |
| `PUT /users/**` | ❌ (403) | ✅ | → redirect to `/auth/login` |
| `DELETE /users/**` | ❌ (403) | ✅ | → redirect to `/auth/login` |
| `/logout` (POST) | ✅ | ✅ | → redirect to `/auth/login` |

### 9.1.3. Множественные роли

Пользователь может иметь одновременно обе роли (`ROLE_USER` и `ROLE_ADMIN`). В этом случае ему доступен полный функционал администратора.

## 9.2. Аутентификация

### 9.2.1. Механизм аутентификации

- **Тип**: form-based authentication (Spring Security)
- **Username**: email пользователя
- **Password**: пароль (проверяется через BCryptPasswordEncoder)
- **Login URL**: `/auth/login` (GET — отображение формы)
- **Login processing URL**: `/process_login` (POST — обработка формы)
- **Failure URL**: `/auth/login?error`

### 9.2.2. Процесс аутентификации

1. Пользователь отправляет email и пароль на `/process_login`.
2. Spring Security вызывает `UserDetailsService.loadUserByUsername(email)`.
3. Система ищет пользователя в БД по email (`UserRepository.findByEmail()`).
4. Если пользователь не найден — выбрасывается `UsernameNotFoundException`.
5. Если найден — Spring Security сравнивает введённый пароль с хешем из БД (BCrypt).
6. При совпадении — создаётся аутентифицированная сессия.
7. Вызывается `SuccessUserHandler.onAuthenticationSuccess()` — редирект на `/admin` или `/user`.

### 9.2.3. UserDetailsService

Реализация `UserDetailsService` загружает пользователя по email:

- Выполняет запрос к БД через `UserRepository.findByEmail(email)`.
- Использует EntityGraph для предзагрузки ролей (устранение N+1).
- Если пользователь не найден — `UsernameNotFoundException("User not found!")`.
- Возвращает сущность User (которая реализует UserDetails).

## 9.3. Авторизация

### 9.3.1. Правила доступа (URL-уровень)

| Правило | Описание |
|---|---|
| `/auth/login`, `/error`, `/js/**` | Доступ без аутентификации (permitAll) |
| `/admin` | Только роль ADMIN (hasRole("ADMIN")) |
| `PATCH /users/**` | Только роль ADMIN |
| `POST /users/**` | Только роль ADMIN |
| `PUT /users/**` | Только роль ADMIN |
| `DELETE /users/**` | Только роль ADMIN |
| Все остальные запросы | Требуют аутентификации (anyRequest().authenticated()) |

### 9.3.2. Редирект при успешном входе

| Роль | URL редиректа |
|---|---|
| Содержит `ROLE_ADMIN` | `/admin` |
| Не содержит `ROLE_ADMIN` (только `ROLE_USER`) | `/user` |

### 9.3.3. Logout

- **URL**: `/logout` (POST)
- **После выхода**: редирект на `/auth/login`

## 9.4. Хранение паролей

| Параметр | Значение |
|---|---|
| Алгоритм хеширования | BCrypt |
| Кодировщик | BCryptPasswordEncoder |
| Применение | При создании и обновлении пользователя пароль хешируется перед сохранением в БД |

## 9.5. CORS-конфигурация

| Параметр | Значение |
|---|---|
| AllowCredentials | true |
| AllowedOrigins | `http://localhost:8080` |
| AllowedHeaders | Authorization, Content-Type, Accept, X-XSRF-TOKEN |
| AllowedMethods | GET, POST, PATCH, DELETE, OPTIONS |
| Применение | Все URL (`/**`) |

## 9.6. CSRF

CSRF-защита **отключена** для поддержки REST API запросов через JavaScript Fetch API без необходимости передачи CSRF-токена.

## 9.7. Статические ресурсы

| URL | Ресурс | Доступ |
|---|---|---|
| `/static/**` | `classpath:/static/` | Без аутентификации (через MvcConfig resource handler) |
| `/js/**` | JavaScript файлы | Без аутентификации (через Spring Security permitAll) |

## 9.8. Состояние аккаунта

Все аккаунты пользователей считаются активными:

| Свойство | Значение | Описание |
|---|---|---|
| `isAccountNonExpired` | true | Аккаунт никогда не истекает |
| `isAccountNonLocked` | true | Аккаунт никогда не блокируется |
| `isCredentialsNonExpired` | true | Учётные данные никогда не истекают |
| `isEnabled` | true | Аккаунт всегда активен |

> В системе не предусмотрены механизмы блокировки, истечения срока действия аккаунта или деактивации пользователей.
