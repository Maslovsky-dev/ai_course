# 7. Описание экранов и пользовательских сценариев

## 7.1. Экраны системы

### 7.1.1. Экран «Вход в систему» (Login page)

**URL**: `/auth/login`
**Доступ**: открыт для всех (без аутентификации)
**Шаблон**: `auth/login.html`

#### Макет

```
┌─────────────────────────────────────────────────┐
│              (фон: WhiteSmoke)                    │
│                                                   │
│            ┌─────────────────────┐                │
│            │  Please sign in     │                │
│            │                     │                │
│            │  [Email address   ] │                │
│            │  [Password        ] │                │
│            │                     │                │
│            │  [   Sign in    ]   │                │
│            │                     │                │
│            │  (сообщение об      │                │
│            │   ошибке, если есть)│                │
│            └─────────────────────┘                │
│                                                   │
└─────────────────────────────────────────────────┘
```

#### Элементы

| Элемент | Тип | Описание |
|---|---|---|
| Заголовок | `<h2>` | Текст "Please sign in" |
| Поле email | `<input type="text">` | Placeholder "Email address", name="username" |
| Поле пароля | `<input type="password">` | Placeholder "Password", name="password" |
| Кнопка входа | `<input type="submit">` | Значение "Sign in", класс `btn btn-primary` |
| Сообщение об ошибке | `<div>` | Отображается при наличии параметра `error` в URL. Текст: "Неправильные имя или пароль", красный цвет. |

#### Поведение

- Форма отправляется методом POST на `/process_login`.
- При успешной аутентификации — редирект на `/admin` (для ADMIN) или `/user` (для USER).
- При ошибке — редирект на `/auth/login?error` с отображением сообщения об ошибке.

### 7.1.2. Экран «Панель администратора» (Admin page)

**URL**: `/admin`
**Доступ**: только роль `ROLE_ADMIN`
**Шаблон**: `admin.html`

#### Макет

```
┌──────────────────────────────────────────────────────────────┐
│  user@email.com with roles: ADMIN USER          [Logout]      │  ← шапка (darkgrey)
├──────────┬───────────────────────────────────────────────────┤
│          │  ┌─ Admin ─┐ ┌─ User ─┐                            │  ← вкладки Admin/User
│  Admin   │  ┌─ Users table ─┐ ┌─ Add user ─┐                 │  ← подвкладки
│  (active)│                                                   │
│  User    │  ┌─────────────────────────────────────────────┐  │
│          │  │  All users                                   │  │
│          │  │  ┌──────┬─────┬─────┬────┬─────┬──────┬─────┐│  │
│          │  │  │ Id   │First│Last │Age │Email│Role  │Edit ││  │
│          │  │  │      │Name │Name │    │     │      │Delete│  │
│          │  │  ├──────┼─────┼─────┼────┼─────┼──────┼─────┤│  │
│          │  │  │ 1    │John │Doe  │ 25 │j@x..│USER  │[Edit]│  │
│          │  │  │      │     │     │    │     │      │[Del] │  │
│          │  │  └──────┴─────┴─────┴────┴─────┴──────┴─────┘│  │
│          │  └─────────────────────────────────────────────┘  │
├──────────┴───────────────────────────────────────────────────┤
│                                                               │
│  (модальные окна Edit / Delete — поверх основного контента)   │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

#### Шапка (верхняя панель)

| Элемент | Описание |
|---|---|
| Email пользователя | Отображается email авторизованного пользователя + текст "with roles:" |
| Роли | Отображаются "ADMIN" (если есть ROLE_ADMIN) и/или "USER" (если есть ROLE_USER) |
| Кнопка Logout | Ссылка `btn-link text-muted`, отправляет POST на `/logout` |
| Фон | `darkgrey`, padding 10px |

#### Левое навигационное меню

| Пункт | Описание |
|---|---|
| **Admin** | Активен по умолчанию. Открывает панель администрирования. |
| **User** | Открывает вкладку с информацией о текущем пользователе. |

Меню реализовано как `nav nav-pills flex-column`, ширина колонки — 2/12 (Bootstrap grid).

#### Вкладка «Admin» — подвкладка «Users table»

**Заголовок**: "Admin panel"

Содержит две подвкладки (nav-tabs):
- **Users table** (активна по умолчанию)
- **Add user**

##### Таблица пользователей

**Заголовок карточки**: "All users"

| Колонка | Описание |
|---|---|
| Id | Идентификатор пользователя |
| First Name | Имя пользователя |
| Last Name | Фамилия пользователя (заголовок колонки ошибочно "First Name", но отображает фамилию) |
| Age | Возраст |
| Email | Email пользователя |
| Role | Роли через запятую (например, "ROLE_USER, ROLE_ADMIN") |
| Edit | Кнопка "Edit" (`btn-info`), открывает модальное окно редактирования |
| Delete | Кнопка "Delete" (`btn-danger`), открывает модальное окно удаления |

Таблица: `table table-hover`, ширина 1200px. Данные загружаются асинхронно через JavaScript (Fetch API → `GET /users/`).

##### Модальное окно «Edit user»

**Заголовок**: "Edit user"

| Поле | Тип | ID | Описание |
|---|---|---|---|
| ID | hidden | `editId` | Скрытое поле с ID пользователя |
| First name | text | `editfirstName` | Имя |
| Last name | text | `editlastName` | Фамилия |
| Age | number | `editage` | Возраст |
| Email | text | `editemail` | Email |
| Password | password | `editpassword` | Пароль |
| Role | select multiple | `editroleSelect` | Выбор ролей (USER, ADMIN), высота 60px |

Кнопки:
- **Close** (`btn-secondary`) — закрывает модальное окно без сохранения
- **Edit** (`btn-primary`) — отправляет PATCH-запрос на `/users/{id}`

##### Модальное окно «Delete user»

**Заголовок**: "Delete user"

| Поле | Тип | ID | Описание |
|---|---|---|---|
| ID | hidden | `deleteId` | Скрытое поле с ID пользователя |
| First name | text (disabled) | `deletefirstName` | Имя (только чтение) |
| Last name | text (disabled) | `deletelastName` | Фамилия (только чтение) |
| Age | number (disabled) | `deleteage` | Возраст (только чтение) |
| Email | text (disabled) | `deleteemail` | Email (только чтение) |
| Password | password (disabled) | `deletepassword` | Пароль (только чтение) |
| Role | select multiple (disabled) | `deleteroleSelect` | Роли (только чтение), высота 60px |

Кнопки:
- **Close** (`btn-secondary`) — закрывает модальное окно
- **Delete** (`btn-primary`) — отправляет DELETE-запрос на `/users/{id}`

#### Вкладка «Admin» — подвкладка «Add user»

**Заголовок карточки**: "Add new user"

Форма центрирована, ширина 350px.

| Поле | Тип | ID | Name | Описание |
|---|---|---|---|---|
| First name | text | `firstName` | `firstName` | Имя |
| Last name | text | `lastName` | `lastName` | Фамилия |
| Age | number | `age` | `age` | Возраст |
| Email | text | `email` | `email` | Email |
| Password | password | `Password` | `Password` | Пароль |
| Role | select multiple | `roleSelect` | `roleSelect` | Выбор ролей, высота 60px |

Кнопка отправки: "Add new user" (`btn-success`, font-size 20px).

#### Вкладка «User» (на странице администратора)

**Заголовок**: "User information-page"
**Заголовок карточки**: "About user"

Отображает таблицу с данными текущего авторизованного пользователя (передаётся через модель `authUser`).

| Колонка | Описание |
|---|---|
| Id | ID авторизованного пользователя |
| First Name | Имя |
| Last Name | Фамилия (заголовок колонки "First Name" — ошибка в UI) |
| Age | Возраст |
| Email | Email |
| Role | "ADMIN" и/или "USER" |

Таблица: `table table-hover`, ширина 1000px. Колонки имеют фиксированную ширину с `text-overflow: ellipsis`.

### 7.1.3. Экран «Страница пользователя» (User page)

**URL**: `/user`
**Доступ**: все аутентифицированные пользователи
**Шаблон**: `user.html`

#### Макет

```
┌──────────────────────────────────────────────────────────────┐
│  user@email.com with roles: USER                [Logout]      │  ← шапка (darkgrey)
├──────────┬───────────────────────────────────────────────────┤
│          │  User information-page                             │
│  User    │  About user                                        │
│  (active)│  ┌──────┬─────┬─────┬────┬─────┬──────┐            │
│          │  │ Id   │First│Last │Age │Email│Role  │            │
│          │  ├──────┼─────┼─────┼────┼─────┼──────┤            │
│          │  │ 1    │John │Doe  │ 25 │j@x..│USER  │            │
│          │  └──────┴─────┴─────┴────┴─────┴──────┘            │
└──────────┴───────────────────────────────────────────────────┘
```

#### Элементы

Идентичны вкладке "User" на странице администратора, но:
- Левое меню содержит только один пункт "User" (активен).
- Нет вкладок администрирования.
- Данные передаются через модель Thymeleaf (`authUser`).

## 7.2. Пользовательские сценарии

### Сценарий UC-1: Вход в систему

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Пользователь открывает браузер и переходит на `http://localhost:8080/` | Spring Security перенаправляет на `/auth/login` |
| 2 | Отображается страница входа с формой | Поля email и пароля пустые |
| 3 | Пользователь вводит email и пароль, нажимает "Sign in" | Форма отправляется POST на `/process_login` |
| 4a | Учётные данные верны, пользователь имеет роль ADMIN | Редирект на `/admin` |
| 4b | Учётные данные верны, пользователь имеет роль USER (без ADMIN) | Редирект на `/user` |
| 4c | Учётные данные неверны | Редирект на `/auth/login?error`, отображается "Неправильные имя или пароль" |

### Сценарий UC-2: Просмотр списка пользователей (Администратор)

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Администратор входит в систему | Редирект на `/admin` |
| 2 | Страница загружается, активна вкладка "Users table" | JavaScript выполняет `fetch('GET /users/')` |
| 3 | REST API возвращает JSON-массив пользователей | JavaScript рендерит таблицу пользователей |
| 4 | В таблице отображаются все пользователи с их данными и ролями | Каждая строка содержит кнопки "Edit" и "Delete" |

### Сценарий UC-3: Создание нового пользователя (Администратор)

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Администратор нажимает вкладку "Add user" | Отображается форма создания пользователя |
| 2 | Заполняет поля: First name, Last name, Age, Email, Password, выбирает роли | Все поля заполнены |
| 3 | Нажимает кнопку "Add new user" | JavaScript собирает данные формы в JSON и отправляет POST на `/users/` |
| 4a | Данные валидны, email уникален | Сервер создаёт пользователя, возвращает HTTP 200. JavaScript обновляет таблицу и переключает на вкладку "Users table" |
| 4b | Данные невалидны (нарушены ограничения) | Сервер возвращает HTTP 400 с описанием ошибок. В консоль выводится ошибка |
| 4c | Email уже существует | Сервер возвращает HTTP 400 с сообщением "Человек с такой почтой уже существует" |

### Сценарий UC-4: Редактирование пользователя (Администратор)

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Администратор нажимает кнопку "Edit" в строке таблицы | JavaScript определяет ID пользователя из строки |
| 2 | JavaScript отправляет `GET /users/{id}` | Сервер возвращает данные пользователя в JSON |
| 3 | Открывается модальное окно "Edit user" с предзаполненными полями | Все поля заполнены, роли выделены в select |
| 4 | Администратор изменяет данные и нажимает "Edit" | JavaScript отправляет PATCH на `/users/{id}` с обновлёнными данными |
| 5a | Данные валидны | Сервер обновляет пользователя, возвращает HTTP 200. Модальное окно закрывается, таблица обновляется |
| 5b | Данные невалидны | Сервер возвращает HTTP 400. В alert отображается "Error updating user" |

### Сценарий UC-5: Удаление пользователя (Администратор)

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Администратор нажимает кнопку "Delete" в строке таблицы | JavaScript определяет ID пользователя из строки |
| 2 | JavaScript отправляет `GET /users/{id}` | Сервер возвращает данные пользователя в JSON |
| 3 | Открывается модальное окно "Delete user" с предзаполненными полями (все поля disabled) | Поля заполнены, роли выделены |
| 4 | Администратор нажимает "Delete" | JavaScript отправляет DELETE на `/users/{id}` |
| 5a | Удаление успешно | Сервер возвращает HTTP 200. Модальное окно закрывается, таблица обновляется |
| 5b | Ошибка удаления | В alert отображается "Error updating user" |

### Сценарий UC-6: Просмотр собственной информации (Пользователь)

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Пользователь с ролью USER входит в систему | Редирект на `/user` |
| 2 | Страница загружается | Thymeleaf рендерит таблицу с данными авторизованного пользователя |
| 3 | В таблице отображаются: ID, имя, фамилия, возраст, email, роли | Данные берутся из модели `authUser` |

### Сценарий UC-7: Просмотр собственной информации (Администратор)

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Администратор входит в систему | Редирект на `/admin` |
| 2 | Нажимает "User" в левом навигационном меню | Отображается вкладка "User information-page" |
| 3 | В таблице отображаются данные авторизованного администратора | ID, имя, фамилия, возраст, email, роли |

### Сценарий UC-8: Выход из системы

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Пользователь нажимает "Logout" в верхней панели | Форма отправляет POST на `/logout` |
| 2 | Spring Security завершает сессию | Редирект на `/auth/login` |
| 3 | Отображается страница входа | Поля пустые, готов к новому входу |

### Сценарий UC-9: Попытка несанкционированного доступа

| Шаг | Действие | Результат |
|---|---|---|
| 1 | Неаутентифицированный пользователь переходит на `/admin` или `/user` | Spring Security перенаправляет на `/auth/login` |
| 2 | Пользователь с ролью USER переходит на `/admin` | Spring Security возвращает ошибку 403 Forbidden |
| 3 | Пользователь с ролью USER отправляет POST на `/users/` | Spring Security возвращает ошибку 403 Forbidden |
