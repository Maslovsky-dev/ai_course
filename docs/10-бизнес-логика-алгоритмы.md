# 10. Бизнес-логика и алгоритмы обработки данных

## 10.1. Управление пользователями

### 10.1.1. Создание пользователя

**Алгоритм**:

1. Клиент отправляет POST-запрос на `/users/` с JSON-телом (UserDTO).
2. REST-контроллер десериализует JSON в UserDTO.
3. Выполняется Bean Validation (аннотации на полях UserDTO).
4. UserDTO преобразуется в сущность User через ModelMapper.
5. Выполняется кастомная валидация уникальности email:
   - Вызывается `UserDetailsServiceImpl.loadUserByUsername(email)`.
   - Если пользователь с таким email найден — в `BindingResult` добавляется ошибка: "Человек с такой почтой уже существует".
6. Если есть ошибки валидации (Bean Validation или кастомные):
   - Все ошибки объединяются в строку формата `поле - сообщение; поле - сообщение`.
   - Выбрасывается `UserNotCreatedException` с этой строкой.
   - Возвращается HTTP 400 с UserErrorResponse.
7. Если ошибок нет:
   - Пароль хешируется через `BCryptPasswordEncoder.encode()`.
   - Пользователь сохраняется в БД через `UserRepository.saveAndFlush()`.
   - Возвращается HTTP 200.

```
[POST /users/] → [Десериализация JSON → UserDTO]
    → [Bean Validation]
    → [ModelMapper: UserDTO → User]
    → [UserValidator: проверка уникальности email]
    → [Ошибки?] ──Да──→ [HTTP 400 + UserErrorResponse]
    │
    Нет
    ↓
    → [BCrypt: хеширование пароля]
    → [UserRepository.saveAndFlush()]
    → [HTTP 200]
```

### 10.1.2. Обновление пользователя

**Алгоритм**:

1. Клиент отправляет PATCH-запрос на `/users/{id}` с JSON-телом (UserDTO).
2. REST-контроллер десериализует JSON в UserDTO.
3. Выполняется Bean Validation (аннотации на полях UserDTO).
4. Если есть ошибки валидации:
   - Все ошибки объединяются в строку формата `поле - сообщение; поле - сообщение`.
   - Выбрасывается `UserNotCreatedException`.
   - Возвращается HTTP 400 с UserErrorResponse.
5. Если ошибок нет:
   - UserDTO преобразуется в сущность User через ModelMapper.
   - Пароль хешируется через `BCryptPasswordEncoder.encode()`.
   - Пользователь сохраняется в БД через `UserRepository.saveAndFlush()` (обновляется существующая запись, если ID совпадает).
   - Возвращается HTTP 200.

> **Отличие от создания**: при обновлении **не выполняется** проверка уникальности email.

```
[PATCH /users/{id}] → [Десериализация JSON → UserDTO]
    → [Bean Validation]
    → [Ошибки?] ──Да──→ [HTTP 400 + UserErrorResponse]
    │
    Нет
    ↓
    → [ModelMapper: UserDTO → User]
    → [BCrypt: хеширование пароля]
    → [UserRepository.saveAndFlush()]
    → [HTTP 200]
```

### 10.1.3. Удаление пользователя

**Алгоритм**:

1. Клиент отправляет DELETE-запрос на `/users/{id}`.
2. REST-контроллер вызывает `UserService.deleteById(id)`.
3. `UserRepository.deleteById(id)` удаляет пользователя из БД.
4. Возвращается HTTP 200.

> Удаление не проверяет существование пользователя. Если ID не найден, поведение зависит от реализации `deleteById` (Spring Data JPA выбрасывает `EmptyResultDataAccessException`).

### 10.1.4. Получение всех пользователей

**Алгоритм**:

1. Клиент отправляет GET-запрос на `/users/`.
2. REST-контроллер вызывает `UserService.findAll()`.
3. `UserRepository.findAll()` выполняет запрос с EntityGraph (предзагрузка ролей).
4. Список сущностей User преобразуется в список UserDTO через ModelMapper.
5. Возвращается HTTP 200 с JSON-массивом.

### 10.1.5. Получение пользователя по ID

**Алгоритм**:

1. Клиент отправляет GET-запрос на `/users/{id}`.
2. REST-контроллер вызывает `UserService.findById(id)`.
3. `UserRepository.findById(id)` выполняет запрос с EntityGraph.
4. Если пользователь найден — преобразуется в UserDTO через ModelMapper, возвращается HTTP 200.
5. Если не найден — `Optional.orElseThrow(UserNotFoundException)` выбрасывает исключение.
6. Обработчик исключения возвращает HTTP 404 с UserErrorResponse (сообщение: "User with this id wasn't found").

## 10.2. Аутентификация

### 10.2.1. Алгоритм входа

1. Пользователь вводит email и пароль на форме входа.
2. Форма отправляется POST на `/process_login`.
3. Spring Security вызывает `UserDetailsServiceImpl.loadUserByUsername(email)`.
4. `UserRepository.findByEmail(email)` ищет пользователя в БД (с EntityGraph для ролей).
5. Если пользователь не найден — `UsernameNotFoundException("User not found!")`.
6. Spring Security сравнивает введённый пароль с хешем из БД через `BCryptPasswordEncoder.matches()`.
7. При успехе — `SuccessUserHandler.onAuthenticationSuccess()`:
   - Получает множество ролей из `Authentication.getAuthorities()`.
   - Если содержит `ROLE_ADMIN` — редирект на `/admin`.
   - Иначе — редирект на `/user`.
8. При неудаче — редирект на `/auth/login?error`.

## 10.3. Клиентская логика (JavaScript)

### 10.3.1. Загрузка таблицы пользователей

**Алгоритм** (функция `getAllUsers`):

1. Выполняется `fetch('http://localhost:8080/users/')`.
2. Ответ парсится как JSON.
3. Для каждого пользователя создаётся объект `User` (JS-класс).
4. Формируется HTML-строка с строками таблицы:
   - Колонки: ID, firstName, lastName, age, email, роли (через запятую), кнопка Edit, кнопка Delete.
   - Роли: `user.roles.map(role => role.roleName).join(", ")`.
5. HTML вставляется в `tbody#usersTable`.

### 10.3.2. Создание пользователя (клиент)

**Алгоритм** (функция `newUser`):

1. Перехватывается событие submit формы `#newUserForm`.
2. Создаётся объект User с полями из формы:
   - `firstName`, `lastName`, `age`, `email`, `password` (из поля `Password`).
   - `roles`: массив объектов `{ roleName: option.value }` из выбранных опций `#roleSelect`.
3. Выполняется `fetch('POST /users/')` с JSON-телом.
4. При успехе:
   - В консоль выводится "New user created!".
   - Вызывается `getAllUsers()` для обновления таблицы.
   - Программно кликается вкладка `#tab1-tab` (переход на "Users table").
5. При ошибке — вывод в консоль.

### 10.3.3. Редактирование пользователя (клиент)

**Алгоритм**:

1. Делегирование событий: клик по кнопке в `#usersTable`.
2. Определение ID: `row.cells[0].textContent` (первая ячейка = ID), преобразование в Number.
3. Вызов `getUser(id, callback)` — `fetch('GET /users/{id}')`.
4. Если кнопка имеет класс `btn-info` (Edit):
   - Вызывается `editProcessing(user, id)`.
   - Заполняются поля модального окна `#editModal` данными пользователя.
   - Сбрасывается выбор ролей, затем выделяются роли пользователя.
   - Модальное окно показывается через `bootstrap.Modal.show()`.
5. При клике кнопки "Edit" в модальном окне:
   - Собирается объект `updatedUser` из полей модального окна.
   - `roles`: массив `{ roleName: option.value }` из выбранных опций.
   - Выполняется `fetch('PATCH /users/{id}')` с JSON-телом.
   - При успехе — модальное окно скрывается, таблица обновляется.
   - При ошибке — `alert('Error updating user')`.

### 10.3.4. Удаление пользователя (клиент)

**Алгоритм**:

1. При клике кнопки с классом `btn-danger` (Delete) в строке таблицы:
   - Вызывается `deleteProcessing(user, id)`.
   - Заполняются поля модального окна `#deleteModal` данными пользователя (все поля disabled).
   - Сбрасывается и устанавливается выбор ролей.
   - Модальное окно показывается.
2. При клике кнопки "Delete" в модальном окне:
   - Получается ID из скрытого поля `#deleteId`.
   - Выполняется `fetch('DELETE /users/{id}')`.
   - При успехе — модальное окно скрывается, таблица обновляется.
   - При ошибке — `alert('Error updating user')`.

## 10.4. Маппинг DTO ↔ Сущности

### 10.4.1. UserDTO → User

Выполняется через `ModelMapper.map(userDTO, User.class)`. Автоматически маппятся:
- `id` → `id`
- `firstName` → `firstName`
- `lastName` → `lastName`
- `age` → `age`
- `email` → `email`
- `password` → `password`
- `Set<RoleDTO>` → `Set<Role>` (по полю `roleName`)

### 10.4.2. User → UserDTO

Выполняется через `ModelMapper.map(user, UserDTO.class)`. Автоматически маппятся все поля, включая `Set<Role>` → `Set<RoleDTO>`.

## 10.5. Транзакционность

Все методы сервисного слоя аннотированы `@Transactional`:

| Сервис | Метод | Транзакция |
|---|---|---|
| UserServiceImpl | `findAll()` | @Transactional |
| UserServiceImpl | `findById()` | @Transactional |
| UserServiceImpl | `save()` | @Transactional |
| UserServiceImpl | `deleteById()` | @Transactional |
| RoleServiceImpl | `findAll()` | @Transactional |
| RoleServiceImpl | `findById()` | @Transactional |
| RoleServiceImpl | `save()` | @Transactional |
| RoleServiceImpl | `deleteById()` | @Transactional |

> Используется `javax.transaction.Transactional` (JTA), а не Spring-аннотация.

## 10.6. Оптимизация запросов к БД

### EntityGraph

Все методы `UserRepository` используют `@EntityGraph(attributePaths = {"roles"})` для предзагрузки ролей в одном SQL-запросе (JOIN), устраняя проблему N+1:

| Метод | EntityGraph |
|---|---|
| `saveAndFlush(User)` | ✅ |
| `findById(Long)` | ✅ |
| `findByEmail(String)` | ✅ |
| `findAll()` | ✅ |

### FetchType

Связь User → Roles использует `FetchType.LAZY`. При прямых запросах к репозиторию EntityGraph обеспечивает загрузку ролей в том же запросе.

## 10.7. Cascade-операции

Связь User → Roles: `CascadeType.MERGE`.

- При сохранении (merge) пользователя роли также мерджатся.
- При удалении пользователя записи из связующей таблицы `user_role` удаляются автоматически (через JOIN).
- Сами роли не удаляются (нет `CascadeType.REMOVE`).
