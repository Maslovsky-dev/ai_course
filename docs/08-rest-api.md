# 8. Описание REST API

## 8.1. Общая информация

| Параметр | Значение |
|---|---|
| Базовый URL | `http://localhost:8080` |
| Формат запросов | JSON (`Content-Type: application/json`) |
| Формат ответов | JSON |
| Аутентификация | Spring Security (form-based, сессия) |
| CSRF | Отключён |

## 8.2. Эндпоинты

### 8.2.1. Получение всех пользователей

| Параметр | Значение |
|---|---|
| **Метод** | `GET` |
| **URL** | `/users/` |
| **Доступ** | Все аутентифицированные пользователи |
| **Тело запроса** | — |
| **Успешный ответ** | HTTP 200, JSON-массив объектов UserDTO |

#### Пример ответа

```json
[
  {
    "id": 1,
    "firstName": "John",
    "lastName": "Doe",
    "age": 25,
    "email": "john@example.com",
    "password": "$2a$10$...",
    "roles": [
      { "roleName": "ROLE_USER" }
    ]
  },
  {
    "id": 2,
    "firstName": "Jane",
    "lastName": "Smith",
    "age": 30,
    "email": "jane@example.com",
    "password": "$2a$10$...",
    "roles": [
      { "roleName": "ROLE_USER" },
      { "roleName": "ROLE_ADMIN" }
    ]
  }
]
```

### 8.2.2. Получение пользователя по ID

| Параметр | Значение |
|---|---|
| **Метод** | `GET` |
| **URL** | `/users/{id}` |
| **Доступ** | Все аутентифицированные пользователи |
| **Path-параметры** | `id` (Long) — идентификатор пользователя |
| **Тело запроса** | — |
| **Успешный ответ** | HTTP 200, JSON-объект UserDTO |
| **Ответ при ненайденном ID** | HTTP 404, JSON-объект UserErrorResponse |

#### Пример успешного ответа

```json
{
  "id": 1,
  "firstName": "John",
  "lastName": "Doe",
  "age": 25,
  "email": "john@example.com",
  "password": "$2a$10$...",
  "roles": [
    { "roleName": "ROLE_USER" }
  ]
}
```

#### Пример ответа при ошибке 404

```json
{
  "message": "User with this id wasn't found",
  "timestamp": 1700000000000
}
```

### 8.2.3. Создание пользователя

| Параметр | Значение |
|---|---|
| **Метод** | `POST` |
| **URL** | `/users/` |
| **Доступ** | Только роль `ROLE_ADMIN` |
| **Тело запроса** | JSON-объект UserDTO (с валидацией) |
| **Успешный ответ** | HTTP 200 (пустое тело) |
| **Ответ при ошибке валидации** | HTTP 400, JSON-объект UserErrorResponse |

#### Тело запроса

```json
{
  "firstName": "John",
  "lastName": "Doe",
  "age": 25,
  "email": "john@example.com",
  "password": "mypassword",
  "roles": [
    { "roleName": "ROLE_USER" }
  ]
}
```

#### Алгоритм обработки

1. Тело запроса десериализуется в UserDTO и проходит Bean Validation.
2. UserDTO преобразуется в сущность User.
3. Выполняется дополнительная валидация: проверка уникальности email (через UserValidator).
4. Если есть ошибки валидации — выбрасывается исключение, возвращается HTTP 400.
5. Пароль хешируется с помощью BCrypt.
6. Пользователь сохраняется в БД (saveAndFlush).
7. Возвращается HTTP 200.

#### Пример ответа при ошибке валидации

```json
{
  "message": "email - Человек с такой почтой уже существует",
  "timestamp": 1700000000000
}
```

> Сообщения об ошибках валидации объединяются через точку с запятой (`;`). Формат: `поле - сообщение`.

---

### 8.2.4. Обновление пользователя

| Параметр | Значение |
|---|---|
| **Метод** | `PATCH` |
| **URL** | `/users/{id}` |
| **Доступ** | Только роль `ROLE_ADMIN` |
| **Path-параметры** | `id` (Long) — идентификатор пользователя |
| **Тело запроса** | JSON-объект UserDTO (с валидацией) |
| **Успешный ответ** | HTTP 200 (пустое тело) |
| **Ответ при ошибке валидации** | HTTP 400, JSON-объект UserErrorResponse |

#### Тело запроса

```json
{
  "id": 1,
  "firstName": "John",
  "lastName": "Doe",
  "age": 26,
  "email": "john.updated@example.com",
  "password": "newpassword",
  "roles": [
    { "roleName": "ROLE_USER" },
    { "roleName": "ROLE_ADMIN" }
  ]
}
```

#### Алгоритм обработки

1. Тело запроса десериализуется в UserDTO и проходит Bean Validation.
2. Если есть ошибки валидации — выбрасывается исключение, возвращается HTTP 400.
3. UserDTO преобразуется в сущность User.
4. Пароль хешируется с помощью BCrypt.
5. Пользователь сохраняется в БД (saveAndFlush — обновляется, если ID существует, или создаётся новый).
6. Возвращается HTTP 200.

> **Примечание**: При обновлении не выполняется проверка уникальности email (в отличие от создания).

---

### 8.2.5. Удаление пользователя

| Параметр | Значение |
|---|---|
| **Метод** | `DELETE` |
| **URL** | `/users/{id}` |
| **Доступ** | Только роль `ROLE_ADMIN` |
| **Path-параметры** | `id` (Long) — идентификатор пользователя |
| **Тело запроса** | — |
| **Успешный ответ** | HTTP 200 (пустое тело) |

#### Алгоритм обработки

1. Вызывается метод удаления пользователя по ID.
2. Пользователь удаляется из БД (deleteById).
3. Возвращается HTTP 200.

## 8.3. Сводная таблица эндпоинтов

| Метод | URL | Доступ | Описание |
|---|---|---|---|
| `GET` | `/users/` | Auth (любая роль) | Получить всех пользователей |
| `GET` | `/users/{id}` | Auth (любая роль) | Получить пользователя по ID |
| `POST` | `/users/` | ADMIN | Создать пользователя |
| `PATCH` | `/users/{id}` | ADMIN | Обновить пользователя |
| `DELETE` | `/users/{id}` | ADMIN | Удалить пользователя |

## 8.4. Структура UserDTO (JSON)

| Поле | Тип JSON | Описание |
|---|---|---|
| `id` | number (Long) | Идентификатор (null при создании) |
| `firstName` | string | Имя (2–30 символов, не пусто) |
| `lastName` | string | Фамилия (2–30 символов, не пусто) |
| `age` | number (int) | Возраст (18–150, не null) |
| `email` | string | Email (2–100 символов, не пусто) |
| `password` | string | Пароль (не пусто) |
| `roles` | array of RoleDTO | Массив ролей (не пустой) |

## 8.5. Структура RoleDTO (JSON)

| Поле | Тип JSON | Описание |
|---|---|---|
| `roleName` | string | Название роли (например, "ROLE_USER") |

## 8.6. Структура UserErrorResponse (JSON)

| Поле | Тип JSON | Описание |
|---|---|---|
| `message` | string | Текст ошибки |
| `timestamp` | number (long) | Временная метка (миллисекунды с epoch) |

## 8.7. Коды ответов

| Код | Описание | Когда возвращается |
|---|---|---|
| 200 OK | Успешный запрос | Все успешные операции |
| 400 Bad Request | Ошибка валидации | Нарушены ограничения полей при создании/обновлении; дублирование email при создании |
| 403 Forbidden | Доступ запрещён | Неаутентифицированный или недостаточно прав (например, USER вызывает POST/PATCH/DELETE) |
| 404 Not Found | Пользователь не найден | Запрос пользователя по несуществующему ID (GET) |
