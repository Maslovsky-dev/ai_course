## 2.1. Общая архитектура

## 2.2. Уровни приложения

Перед началом работы необходимо запустить локальную инфраструктуру PostgreSQL 16 и отдельный контейнер Liquibase. Compose-файл находится в каталоге `infra`, поэтому запуск выполняется командами `cd infra` и `docker compose up -d`. После запуска контейнеров база данных полностью готова к работе: Docker Compose создаёт пользователя PostgreSQL, базу данных `ai_course`, схему приложения `admin_panel`, запускает Liquibase, применяет миграции и заполняет базу тестовыми данными. SQL-скрипты вручную выполнять не требуется. Java-приложение не подключает Liquibase, не запускает миграции средствами Spring Boot и работает только со схемой приложения `admin_panel`.

### 2.2.1. Уровень контроллеров (Controller Layer)

Обрабатывает входящие HTTP-запросы и возвращает представления или JSON-ответы. Разделён на три типа контроллеров:

- **Контроллер аутентификации** — отображает страницу входа.
- **Контроллер представлений (View Controller)** — отдаёт серверно-рендеримые HTML-страницы для административной и пользовательской панелей.
- **REST-контроллер** — обрабатывает REST API запросы для CRUD-операций над пользователями.

### 2.2.2. Уровень сервисов (Service Layer)

Содержит бизнес-логику приложения. Каждый сервис инкапсулирует операции над сущностями:

- **Сервис пользователей** — управление пользователями (создание, получение, удаление), хеширование паролей.
- **Сервис ролей** — управление ролями.
- **Сервис UserDetails** — интеграция с Spring Security для загрузки пользователя по email.

### 2.2.3. Уровень репозиториев (Repository Layer)

Обеспечивает доступ к данным через Spring Data JPA. Репозитории расширяют `JpaRepository` и предоставляют стандартные CRUD-операции, а также кастомные методы поиска.

### 2.2.4. Уровень безопасности (Security Layer)

Конфигурация Spring Security: настройка правил доступа, аутентификация, обработка успешного входа, шифрование паролей.

## 2.3. Паттерны проектирования

| Паттерн | Применение |
|---|---|
| **MVC (Model-View-Controller)** | Разделение на контроллеры, модели и представления |
| **DTO (Data Transfer Object)** | Передача данных между клиентом и сервером через UserDTO и RoleDTO |
| **Service Layer** | Выделение бизнес-логики в отдельный слой |
| **Repository** | Абстракция доступа к данным |
| **Dependency Injection** | Внедрение зависимостей через конструкторы (Spring DI) |
| **Entity Graph** | Оптимизация загрузки связей (roles) для устранения N+1 проблемы |
| **Object Mapper** | Преобразование между сущностями и DTO (ModelMapper) |

## 2.6. Поток данных

1. **Вход в систему**: браузер → `GET /auth/login` → Thymeleaf рендерит форму → пользователь вводит email/пароль → `POST /process_login` → Spring Security аутентифицирует → `SuccessUserHandler` редиректит на `/admin` или `/user`.

2. **Просмотр списка пользователей (админ)**: браузер → `GET /admin` → Thymeleaf рендерит admin.html → JavaScript `fetch('GET /users/')` → REST-контроллер → UserService → UserRepository → PostgreSQL → JSON-ответ → JavaScript рендерит таблицу.

3. **Создание пользователя**: браузер (JavaScript) → `POST /users/` с JSON → REST-контроллер валидирует → UserService хеширует пароль → UserRepository → PostgreSQL → HTTP 200 → JavaScript обновляет таблицу.

4. **Редактирование пользователя**: браузер (JavaScript) → `GET /users/{id}` для получения данных → заполнение модального окна → `PATCH /users/{id}` с JSON → REST-контроллер валидирует → UserService → UserRepository → PostgreSQL → HTTP 200.

5. **Удаление пользователя**: браузер (JavaScript) → `GET /users/{id}` для получения данных → заполнение модального окна → `DELETE /users/{id}` → UserService → UserRepository → PostgreSQL → HTTP 200.
