# 12. Конфигурация приложения и критерии приёмки

## 12.1. Требования к конфигурации приложения

### 12.1.1. Параметры базы данных

| Параметр | Значение | Описание |
|---|---|---|
| `spring.datasource.driver-class-name` | `org.postgresql.Driver` | JDBC-драйвер PostgreSQL |
| `spring.datasource.url` | `jdbc:postgresql://localhost:5432/ai_course` | URL подключения к БД |
| `spring.datasource.username` | `postgres` | Имя пользователя БД |
| `spring.datasource.password` | `postgres` | Пароль пользователя БД |

> **Примечание**: При реализации значения username и password должны быть настраиваемыми (через переменные окружения или external config), а не захардкоженными.

### 12.1.8. Инициализация БД

Перед началом работы необходимо запустить локальную инфраструктуру:

- PostgreSQL 16
- отдельный контейнер Liquibase

Compose-файл находится в каталоге `infra`, поэтому запуск выполняется из этого каталога:

```bash
cd infra
docker compose up -d
```

После запуска контейнеров база данных полностью готова к работе. Docker Compose автоматически создаёт пользователя PostgreSQL, базу данных `ai_course`, отдельную схему приложения `admin_panel`, запускает контейнер Liquibase, применяет все миграции и заполняет базу тестовыми данными. Никаких SQL-скриптов вручную выполнять не требуется.

Студент не должен создавать пользователя, базу данных, схему, выполнять SQL-скрипты или импортировать данные вручную.

Liquibase не должен использоваться внутри Java-проекта: приложение не подключает Liquibase, не хранит файлы миграций в ресурсах и не запускает миграции средствами Spring Boot.

Во всех примерах `application.properties`, `application.yml`, Hibernate и JPA должна быть явно указана схема приложения `admin_panel`. Использование схемы `public` не допускается.

### 12.1.9. Сборка и запуск

| Команда | Описание |
|---|---|
| `cd infra && docker compose up -d` | Запуск PostgreSQL 16 и отдельного контейнера Liquibase из каталога с compose-файлом |
| `cd infra && docker compose ps` | Проверка состояния контейнеров |
| `cd infra && docker compose logs liquibase` | Проверка результата применения миграций |
| `cd infra && docker compose down` | Остановка локальной инфраструктуры |
| `mvn spring-boot:run` | Запуск приложения в режиме разработки |
| `mvn package` | Сборка JAR-файла |
| `java -jar target/spring-boot_security-demo-0.0.1-SNAPSHOT.jar` | Запуск собранного JAR |
| `./mvnw spring-boot:run` | Запуск через Maven wrapper |

### 12.1.10. Параметры окружения

| Параметр | Требование |
|---|---|

## 12.2. Критерии приёмки

### 12.2.1. Аутентификация и авторизация

| Критерий | Описание |
|---|---|
| AC-1 | При переходе на любой защищённый URL без аутентификации пользователь перенаправляется на `/auth/login`. |
| AC-2 | Страница входа отображает форму с полями email и пароля, кнопкой "Sign in". |
| AC-3 | При вводе верных учётных данных пользователь с ролью ADMIN перенаправляется на `/admin`. |
| AC-4 | При вводе верных учётных данных пользователь с ролью USER (без ADMIN) перенаправляется на `/user`. |
| AC-5 | При вводе неверных учётных данных отображается сообщение "Неправильные имя или пароль". |
| AC-6 | При нажатии "Logout" пользователь выходит из системы и перенаправляется на `/auth/login`. |
| AC-7 | Пользователь с ролью USER при попытке доступа к `/admin` получает ошибку 403. |
| AC-8 | Пользователь с ролью USER при попытке отправить POST/PATCH/DELETE на `/users/` получает ошибку 403. |
| AC-9 | Пароли в БД хранятся в зашифрованном виде (BCrypt), не в открытом виде. |

### 12.2.2. Панель администратора

| Критерий | Описание |
|---|---|
| AC-10 | Страница `/admin` отображает шапку с email и ролями текущего пользователя, кнопку Logout. |
| AC-11 | Левое меню содержит пункты "Admin" и "User". |
| AC-12 | Вкладка "Users table" отображает таблицу всех пользователей с колонками: ID, First Name, Last Name, Age, Email, Role, Edit, Delete. |
| AC-13 | Таблица загружается асинхронно через REST API при загрузке страницы. |
| AC-14 | В колонке "Role" роли отображаются через запятую (например, "ROLE_USER, ROLE_ADMIN"). |
| AC-15 | Каждая строка таблицы содержит кнопки "Edit" (синяя) и "Delete" (красная). |
| AC-16 | Вкладка "Add user" содержит форму с полями: First name, Last name, Age, Email, Password, Role (multiple select). |
| AC-17 | Multiple select для ролей содержит варианты USER и ADMIN. |
| AC-18 | При нажатии "Add new user" данные отправляются на сервер, таблица обновляется, происходит переход на вкладку "Users table". |
| AC-19 | Вкладка "User" на странице администратора отображает таблицу с данными текущего пользователя. |

| AC-21 | Данные пользователя загружаются асинхронно по ID перед открытием модального окна. |

### 12.2.4. Удаление пользователя

| Критерий | Описание |

| AC-27 | Данные пользователя загружаются асинхронно по ID перед открытием модального окна. |
| AC-28 | При нажатии "Delete" в модальном окне отправляется DELETE-запрос, модальное окно закрывается, таблица обновляется. |

### 12.2.5. Страница пользователя

| Критерий | Описание |
|---|---|
| AC-31 | Страница `/user` отображает шапку с email и ролями, кнопку Logout. |

### 12.2.6. REST API

| Критерий | Описание |
|---|---|
| AC-35 | `GET /users/` возвращает JSON-массив всех пользователей с ролями. |
| AC-36 | `GET /users/{id}` возвращает JSON-объект пользователя. |
| AC-37 | `GET /users/{id}` с несуществующим ID возвращает HTTP 404 с UserErrorResponse. |
| AC-38 | `POST /users/` с валидными данными создаёт пользователя и возвращает HTTP 200. |
| AC-39 | `POST /users/` с невалидными данными возвращает HTTP 400 с описанием ошибок. |
| AC-40 | `POST /users/` с дублирующим email возвращает HTTP 400 с сообщением "Человек с такой почтой уже существует". |
| AC-41 | `PATCH /users/{id}` с валидными данными обновляет пользователя и возвращает HTTP 200. |
| AC-42 | `PATCH /users/{id}` с невалидными данными возвращает HTTP 400 с описанием ошибок. |
| AC-43 | `DELETE /users/{id}` удаляет пользователя и возвращает HTTP 200. |

### 12.2.7. Валидация

| Критерий | Описание |
|---|---|
| AC-44 | Имя (firstName) не может быть пустым, длина 2–30 символов. |
| AC-45 | Фамилия (lastName) не может быть пустой, длина 2–30 символов. |
| AC-46 | Возраст (age) должен быть от 18 до 150. |
| AC-47 | Email не может быть пустым, длина 2–100 символов. |
| AC-48 | Пароль не может быть пустым. |
| AC-49 | Должна быть выбрана хотя бы одна роль. |
| AC-50 | Email должен быть уникален при создании (проверка через БД). |

| Критерий | Описание |
|---|---|
| AC-51 | CORS настроен для origin `http://localhost:8080`. |
| AC-52 | Разрешены HTTP-методы: GET, POST, PATCH, DELETE, OPTIONS. |
| AC-53 | Статические JS-файлы доступны без аутентификации. |

### 12.2.9. База данных

| Критерий | Описание |
|---|---|
| AC-55 | Таблица `users` содержит колонки: id, first_name, last_name, age, email, password. |
| AC-56 | Таблица `roles` содержит колонку role_name (PK). |
| AC-57 | Связующая таблица `user_role` содержит user_id (FK → users.id) и role_name (FK → roles.role_name), составной PK. |
| AC-58 | Hibernate работает в режиме validate (не создаёт и не изменяет схему автоматически). |
| AC-59 | В таблице `roles` существуют записи `ROLE_USER` и `ROLE_ADMIN`. |
| AC-60 | Все таблицы приложения находятся в схеме `admin_panel`; схема `public` не используется. |
| AC-61 | После перехода в каталог `infra` и выполнения `docker compose up -d` пользователь PostgreSQL, база данных, схема, миграции и тестовые данные готовы без ручных действий студента. |
