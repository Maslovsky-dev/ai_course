# 6. Описание сущностей и структуры базы данных

## 6.1. Сущность «Пользователь» (User)

Пользователь системы. Реализует интерфейс `UserDetails` для интеграции с Spring Security.

### Поля

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|
| `id` | BIGINT (Long) | Primary Key, Generated Identity | Уникальный идентификатор пользователя |
| `firstName` | VARCHAR(100) (String) | NotNull, длина 2–30 | Имя пользователя |
| `lastName` | VARCHAR(100) (String) | NotNull, длина 2–30 | Фамилия пользователя |
| `age` | INT (int) | NotNull, min 18, max 150 | Возраст пользователя |
| `email` | VARCHAR(100) (String) | NotNull, длина 2–100 | Email (используется как username для аутентификации) |
| `password` | VARCHAR(255) (String) | NotNull | Пароль (хранится в зашифрованном виде BCrypt) |
| `roles` | Set\<Role\> | NotNull, M:N | Множество ролей пользователя |

### Особенности

- **Имя таблицы**: `users`
- **Username**: в качестве имени пользователя для Spring Security используется поле `email` (метод `getUsername()` возвращает `email`).
- **Связь с ролями**: Many-to-Many через промежуточную таблицу `user_role`, cascade = MERGE, fetch = LAZY.
- **Аккаунт всегда активен**: методы `isAccountNonExpired()`, `isAccountNonLocked()`, `isCredentialsNonExpired()`, `isEnabled()` всегда возвращают `true`.
- **Authorities**: метод `getAuthorities()` возвращает множество ролей пользователя.

## 6.2. Сущность «Роль» (Role)

Роль пользователя в системе. Реализует интерфейс `GrantedAuthority` для интеграции с Spring Security.

### Поля

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|
| `roleName` | VARCHAR(50) (String) | Primary Key | Название роли (например, `ROLE_USER`, `ROLE_ADMIN`) |

### Особенности

- **Имя таблицы**: `roles`
- **Primary Key**: `role_name` (строковый идентификатор)
- **Authority**: метод `getAuthority()` возвращает значение `roleName`.
- **Предопределённые роли**: `ROLE_USER`, `ROLE_ADMIN`.

## 6.3. Структура базы данных

### База данных: `ai_course`

### Схема приложения: `admin_panel`

Все таблицы приложения должны находиться в схеме `admin_panel`. Схема `public` не используется.

### Таблица `users`

| Колонка | Тип | Ограничения |
|---|---|---|
| `id` | BIGINT GENERATED BY DEFAULT AS IDENTITY | PRIMARY KEY |
| `first_name` | VARCHAR(100) | NOT NULL |
| `last_name` | VARCHAR(100) | NOT NULL |
| `age` | INT | NOT NULL |
| `email` | VARCHAR(100) | NOT NULL |
| `password` | VARCHAR(255) | NOT NULL |

> **Примечание**: Фактическая схема должна соответствовать JPA-сущности `User` со всеми полями: `id`, `first_name`, `last_name`, `age`, `email`, `password`.

### Таблица `roles`

| Колонка | Тип | Ограничения |
|---|---|---|
| `role_name` | VARCHAR(50) | PRIMARY KEY |

### Таблица `user_role` (связующая)

| Колонка | Тип | Ограничения |
|---|---|---|
| `user_id` | BIGINT | NOT NULL, FK → users(id) |
| `role_name` | VARCHAR(50) | NOT NULL, FK → roles(role_name) |
| | | PRIMARY KEY (user_id, role_name) |

### ER-диаграмма

```
┌──────────────────┐         ┌──────────────────┐         ┌──────────────────┐
│      users       │         │    user_role     │         │      roles       │
├──────────────────┤         ├──────────────────┤         ├──────────────────┤
│ id (PK)          │◄────────│ user_id (FK)     │────────►│                  │
│ first_name       │         │ role_name (FK)   │────────►│ role_name (PK)   │
│ last_name        │         │ PK(user_id,      │         │                  │
│ age              │         │    role_name)    │         │                  │
│ email            │         └──────────────────┘         └──────────────────┘
│ password         │
└──────────────────┘
```

### Связи

| Связь | Тип | Описание |
|---|---|---|
| User → Role | Many-to-Many | Один пользователь может иметь много ролей; одна роль может принадлежать многим пользователям. Связь через таблицу `user_role`. |
| Role → User | Many-to-Many | Обратная связь. Cascade = MERGE. |

### Инициализация данных

Перед началом работы необходимо запустить локальную инфраструктуру:

- PostgreSQL 16
- отдельный контейнер Liquibase

Compose-файл находится в каталоге `infra`, поэтому запуск выполняется из этого каталога:

```bash
cd infra
docker compose up -d
```

После запуска контейнеров база данных полностью готова к работе. Docker Compose автоматически создаёт пользователя PostgreSQL, базу данных `ai_course`, схему приложения `admin_panel`, запускает контейнер Liquibase, применяет все миграции и заполняет базу тестовыми данными. Никаких SQL-скриптов вручную выполнять не требуется.

При первичной настройке БД миграциями отдельного контейнера Liquibase в схеме `admin_panel` должны быть созданы:

1. **Роли**:
   - `ROLE_USER`
   - `ROLE_ADMIN`

2. **Тестовые пользователи** (опционально, для начального запуска):
   - Пользователь с email `testuser1@example.com`, роль `ROLE_USER`
   - Пользователь с email `testuser2@example.com`, роль `ROLE_USER`
   - Пользователь с email `testuser3@example.com`, роль `ROLE_ADMIN`

> **Важно**: Пароли тестовых пользователей в БД должны храниться в BCrypt-хешированном виде. При реализации необходимо хешировать пароли при инициализации данных через отдельный контейнер Liquibase.

## 6.4. DTO (Data Transfer Objects)

### UserDTO

Используется для передачи данных пользователя через REST API. Содержит те же поля, что и сущность User, но роли представлены как `Set<RoleDTO>`.

| Поле | Тип | Валидация |
|---|---|---|
| `id` | Long | — |
| `firstName` | String | NotEmpty, размер 2–30 |
| `lastName` | String | NotEmpty, размер 2–30 |
| `age` | int | Min 18, Max 150, NotNull |
| `email` | String | NotEmpty, размер 2–100 |
| `password` | String | NotEmpty |
| `roles` | Set\<RoleDTO\> | NotEmpty |

### RoleDTO

| Поле | Тип | Описание |
|---|---|---|
| `roleName` | String | Название роли |

### UserErrorResponse

Используется для возврата информации об ошибках через REST API.

| Поле | Тип | Описание |
|---|---|---|
| `message` | String | Текст ошибки |
| `timestamp` | long | Временная метка ошибки (System.currentTimeMillis()) |

## 6.5. Маппинг между сущностями и DTO

| Направление | Механизм | Описание |
|---|---|---|
| User → UserDTO | ModelMapper | Автоматическое преобразование всех полей, включая Set\<Role\> → Set\<RoleDTO\> |
| UserDTO → User | ModelMapper | Автоматическое преобразование всех полей, включая Set\<RoleDTO\> → Set\<Role\> |
